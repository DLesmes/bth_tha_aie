## Tech Stack Selection
This tech stack is optimized for a deterministic, security-first runtime. It balances fast execution loops with rigorous enterprise-grade access control.

## LLM Provider and Model Choice

* Choice: Google Gemini (gemini-3.5-flash) via langchain-google-genai.
* Rationale: Selected for its low latency, high token throughput, and aggressive pricing structure. Because our architecture relies on an iterative agent loop (Specialist \leftrightarrow Reviewer), we require a model that can run multiple internal evaluations quickly without skyrocketing API costs.
* Alternatives Considered: 
    * OpenAI GPT-4o: Excluded due to higher API costs for multi-agent state loops. 
    * Local Llama-3: Excluded to minimize local computational footprint and infrastructure setup for this demo deployment.

## Embedding Model and Vector Database

* Choice: Hugging Face all-MiniLM-L6-v2 (Local compute) + Hosted Weaviate Cloud Sandbox.
* Rationale: Using all-MiniLM-L6-v2 removes external API latency and cost from our vectorization pipeline. Weaviate v4 provides an exceptionally clean Python client footprint (connect_to_weaviate_cloud), high-speed vector retrieval, and native compatibility with LangChain wrapper classes.
* Alternatives Considered: 
    * Pinecone: Excluded because Weaviate’s object-centric schema architecture structures metadata tagging more natively for downstream integration with Auth0 FGA. 
    * OpenAI Embeddings (text-embedding-3-small): Excluded to prevent network dependency during chunk indexing.

## Orchestration Framework
* Choice: LangGraph.
* Rationale: Standard RAG pipelines are linear Directed Acyclic Graphs (DAGs). This system explicitly requires a cyclic state graph to support a conditional evaluation loop (allowing the Reviewer to pass data back to the Specialist) and the early-exit routing implemented in the Front Agent.
* Alternatives Considered: 
    * Standard LangChain (LCEL) or LlamaIndex: Excluded because while they excel at straight-line data flow, managing complex states, retry limits, and recursive loops in them requires writing messy, error-prone custom boilerplate.

## Document Parsing and Chunking Strategy
* Choice: PyPDFLoader + RecursiveCharacterTextSplitter (Size: 1000, Overlap: 150).
* Rationale: Recursive splitters prioritize splitting by structural elements like paragraphs and sentences rather than raw token limits. This minimizes context fragmentation. It also allows us to inject access metadata tags (doc_id and allowed_role) directly into each chunk's metadata matrix for granular access filtering.
* Alternatives Considered: 
    * CharacterTextSplitter: Excluded because rigid, character-exact cuts break sentences mid-word, resulting in degraded semantic embeddings.

## Backend API and Infrastructure
* Choice: FastAPI + Uvicorn + uv Package Manager.
* Rationale: FastAPI delivers native async capabilities, ultra-fast performance, and automated Pydantic request validation out-of-the-box. Incorporating uv allows for instantaneous dependency mapping and isolated project execution.
* Alternatives Considered: 
    * Flask: Excluded because it lacks native async support and automated schema documentation, making it poorly suited for modern AI application payloads.

## Frontend / Interface Layer
* Choice: Streamlit.
* Rationale: Streamlit allows us to stand up a clean, full-stack conversational dashboard within a single Python script. Crucially, its native session memory structure allows us to build an instant "Identity Session Switcher" sidebar to simulate switching between an admin user and a collaborator user on-the-fly.
* Alternatives Considered: 
    * React / Next.js: Excluded because building a custom web dashboard introduces unnecessary development overhead for a backend-centric RAG capability demo.

## Observability, Logging, and Eval Tooling
* Choice: LangSmith (Graph Tracing) + Programmatic Reviewer Agent Loop.
* Rationale: LangSmith delivers visual tracing of the entire LangGraph execution lifecycle, showing precisely which data payloads are stripped by Auth0 FGA and why. Complementing this, our programmatic Reviewer Agent provides real-time, runtime verification against factual hallucination before returning data to the user.
* Alternatives Considered: 
    * Traditional Logging (ELK / Prometheus): Excluded because basic text dumps do not capture the conversational state step transitions and token usage metadata critical for optimizing LLM workflows.
